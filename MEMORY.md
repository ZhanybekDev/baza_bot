### Двухуровневый ответ: alias-роутинг + project-scoped контекст

- **Решение:** вопрос → keyword fast-path (canned) → регистронезависимый alias-роутер проекта → весь корпус проекта в контекст LLM. Retrieval внутри проекта не нужен — корпус одного ЖК мал.
- **Дата:** 2026-07-16
- **Контекст:** база знаний ~60k токенов, 6 ЖК; классический RAG путал проекты («Алиса» — и ЖК, и колонка).
- **Альтернативы (все отвергнуты по живым замерам):** full-context 69k (галлюцинирует — приписал ДМД2 потолки из ЖК ДНК), наивный RAG top-6 (3 провала из 4), RAG+реранк (сломал ДМД2), семантический роутер проекта (зазор score 0.03).
- **Почему так:** project-scoped дал 4/4 верных ответа; роутинг детерминирован справочником заказчика; худший случай 0.259¢ и 5.28s при лимитах ТЗ 5¢/30s.

- **Решение:** `server/registry/projects.ts` задаёт для 11 проектов алиасы, регион (ЕКБ/МСК/Бали), класс (застройщик/агентство), комиссию. Проект НЕ угадывается из пути файла.

## Известные проблемы / ограничения

- **Env прод-контейнеров ≠ локальный `.env` автоматически.** Каждый ключ надо явно прокинуть в `environment:` нужного сервиса `docker-compose.prod.yml` И положить в серверный `.env`. `SONIOX_API_KEY` выпал из бот-сервиса → голос молча не работал, пока текст работал (2026-07-20). Правило: при добавлении новой внешней интеграции — свериться, что её ключ есть в env-блоке того сервиса, который её вызывает. `:?` в compose даёт fail-fast, если ключ забыт.

- **История git ПЕРЕПИСАНА** (2026-07-18, `git-filter-repo`) — из всех коммитов убраны детали инфраструктуры (IP, tunnel-ID, имена сторонних проектов, SSH-юзер, путь к файлу с паролями). Force-push всех веток. **Следствие:** checkout на сервере `~/baza_bot` диверджнут от `origin` — при следующем обновлении делать `git fetch origin && git reset --hard origin/main` (не `git pull`). Бэкап старой истории — bundle в scratchpad сессии. Реальных секретов в репо не было (проверено).

- **Google/Yandex троттлят 503/429** при частых скачиваниях — в `http.ts` retry 5× с backoff + паузы 800мс между binary-источниками. При первом холодном прогоне может потребоваться 2 запуска (идемпотентность добьёт).
- **Идемпотентность документа — по наличию чанков**, не только по hash: OK-документ с 0 чанков (эмбеддинг упал на прошлом прогоне) считается битым и пересоздаётся. Иначе битый документ пропускался бы навсегда.
- **Сайты Tilda переэмбеддятся каждый прогон** (контент чуть плавает → новый hash). Не дубли (старый документ источника удаляется), но лишний расход. Приемлемо на 2 сайтах.
- **TypeScript 5.9.3, не 7.0**: typescript-eslint 8.x требует TS `<6.1.0`. TS 7 (Go-порт) тулчейн ещё не догнал.
- **Портал `realtors.baza.bz` троттлит 403** при быстрых запросах подряд — ingest портала идёт headed + паузы ~1.2с.
- **Docker build рвёт скачивание крупных Prisma-пакетов** (error 23) и на сервере, и в локальном Docker Desktop — при том что `pnpm install` на хосте работает. Решение: `docker build --network=host`. Для сервера с плохой сетью — собирать образ локально (`--network=host`) и переносить через `docker save`/`load`, а не собирать на сервере. Деплой на сервер использует `docker-compose.deploy.yml` (override с `image:` вместо `build:`, не в git).
- **Деплой-факты:** сервер — SSH через Tailscale `<SERVER_HOST>` (локальный `<SERVER_LAN>` доступен только из домашней сети). Prod-БД `baza_bot_prod_db` (pgvector), порт не публикуется на хост. Наполнение prod — дампом локальной БД (`pg_dump --data-only --disable-triggers`), а не ingest-в-Docker (обходит xvfb-риск). Один токен = один long polling: перед запуском серверного бота остановить локального (иначе 409 Conflict).
- **best.baza.bz блокирует headless** (заглушка BotFAQtor) — парсинг только headed chromium + xvfb.
- **Tailscale на сервере не поднят** — удалённо чинить бота нельзя, только из дома. Поднять до защиты.
- **Кэш bothub недетерминирован** — на него не закладываться.
- **Данные сессии планирования** (индекс, корпус, дерево портала) лежали в `/tmp/.../scratchpad/` — временно. Ingest пересоберёт из источников.
