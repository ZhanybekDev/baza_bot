# Rule: Code Quality Standards

Чеклист качества кода для baza_bot. Применяется на этапе Execute и Self-Review, не на этапе планирования.

---

## 1. Типы везде

- **Python:** full type hints на всех функциях, методах, атрибутах классов. `mypy` (strict) должен проходить чисто.
- **TypeScript:** `strict: true` в `tsconfig.json`. Ноль `any`. Ноль `@ts-ignore` (только `@ts-expect-error` с обязательным комментарием).
- **Generic там где есть реиспользование.** Не на каждую функцию — только когда правда нужно.

---

## 2. Имена

- **Читаемые.** `user_count` лучше `cnt`. `fetch_orders_by_customer` лучше `get_data`.
- **Одно понятие — одно имя.** Не переключайся между `user`/`client`/`customer` если это одна сущность.
- **Никаких магических чисел.** `MAX_RETRIES = 3` константа, а не литерал `3` в трёх местах.
- **Никаких магических строк.** Enum / `typing.Literal` / константа.

---

## 3. Комментарии

**Правило:** комментарий объясняет _почему_, не _что_.

### ❌ Плохо

```python
# Increment counter
counter += 1

# Loop through users
for user in users:
    ...
```

### ✅ Хорошо

```python
# Rate-limited by Stripe — batch of 100 max, see https://stripe.com/docs/rate-limits
for chunk in chunks(orders, 100):
    ...
```

**Когда комментарий нужен:**
- Объяснить rationale неочевидного решения
- Предупредить о ловушке / edge case
- Сослаться на external context (API docs, issue, compliance)

**Когда комментарий не нужен:**
- Описать что делает функция — это делает имя и сигнатура
- Повторить код словами
- Помечать очевидное ("этот loop проходит по списку")

---

## 4. Структура функций

- **Одна функция — одна задача.** Если функция делает "fetch и parse и save" — разбей.
- **Чистые функции где возможно.** Side effects — явно вынести в отдельные функции.
- **Короткие сигнатуры.** Больше 4-5 параметров = передавай объект / dataclass / Pydantic.
- **Возврат данных, а не мутация аргумента.** Если мутируешь — это должно быть очевидно из имени.

---

## 5. Error handling

- **Явно.** Никаких `except:` без типа. `except Exception` — только если точно хочешь поймать всё (и тогда логируй + re-raise).
- **Конкретно.** Доменные исключения с осмысленными именами (`CreatorNotFoundError`, не `Exception("not found")`).
- **На границе системы.** Input validation на границах (Pydantic / Zod), не в глубине бизнес-логики.
- **Не глуши.** Смотри `.claude/rules/no-fallbacks.md`.

---

## 6. Декомпозиция — по смыслу, не по размеру

**Файл в 500-800 строк — это нормально**, если он описывает одну логически цельную концепцию.

Разбивай **только если** выполнено хоть одно:

- Смешаны несколько независимых визуальных / функциональных блоков
- Смешаны разные ответственности (data fetching + бизнес-логика + presentation)
- Дублируется логика в двух местах
- Два разных потока данных в одном файле

**Не разбивай:**
- "Потому что >300 строк" — это не критерий
- "На всякий случай" — YAGNI
- Если после разбиения придётся поднимать 8 props обратно в родителя

---

## 7. Constants и конфиг

- **Никаких хардкод секретов.** Всё через `.env` + settings object (Pydantic `BaseSettings` / `dotenv` / env).
- **Конфиг — одно место.** `config.py` / `settings.ts` / `env.ts`. Не разбросан по файлам.
- **Константы группируй.** Если у тебя 5 MAX_* — они в одном файле `constants.py`, а не по проекту.

---

## 8. Dependencies

- **Минимизируй.** Каждая новая dependency — решение. Не добавляй только чтобы сэкономить 20 строк.
- **Проверяй maintenance.** Библиотека без коммитов за год = риск. Используй Exa (см. `mcp-search.md`) чтобы проверить.
- **Lock files** всегда в git. `package-lock.json`, `poetry.lock`, `yarn.lock`, `Pipfile.lock`.

---

## 9. Async / Concurrency

- **Не мешай sync и async.** Если функция делает I/O — она `async`.
- **`asyncio.gather` / `Promise.all`** для параллельных независимых I/O операций.
- **Никакого `time.sleep()` в async функции** — только `asyncio.sleep()`.
- **Session lifecycle** — один session / connection per request, не shared global.

---

## 10. Tests

- **Unit 100% публичных функций** (кроме очевидно тривиальных getters/setters).
- **Тестируем спецификацию, не реализацию.** Тест должен пройти если ты рефакторишь внутренности.
- **Именование:** `should_[expected]_when_[condition]` или аналогичное.
- **No skip.** Если тест не готов — не коммить; не коммить `test.skip`.

---

## 11. Commits / PRs

- **Маленькие коммиты** с ясным сообщением.
- **Conventional Commits** (`feat:`, `fix:`, `refactor:`, `test:`, `docs:`) — если команда это использует.
- **Один PR — одна тема.** Не смешивай рефакторинг с фичей.

---

## Final Self-Review чеклист

Перед объявлением DONE — пройди:

- [ ] Типы полные, линтер и type-checker чистые
- [ ] Нет `any`, `@ts-ignore`, `except:`
- [ ] Нет `TODO`, `FIXME`, `console.log`, `print`, `debugger`
- [ ] Нет закомментированного кода
- [ ] Нет магических чисел / строк
- [ ] Комментарии только там где объясняют _почему_
- [ ] Ошибки обрабатываются явно
- [ ] Тесты написаны и проходят
- [ ] Критичные flows (ответы бота по 12 тест-кейсам, повторяемый ingest, голосовые сообщения) не сломаны
- [ ] Lint + build + tests локально зелёные

Детальный полный чеклист — в skill `ship-it` (если установлен) или в `self-review.md`.
