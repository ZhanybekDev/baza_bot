# Rule: Self-Review Before DONE

## Правило

Перед тем как объявить задачу DONE — обязательный self-review: перечитай все изменённые файлы целиком, пройди чеклист, запусти локально lint / build / tests. Если хоть один пункт failed — вернись в Execute, почини, пройди чеклист снова.

**Ты — единственный разработчик этого проекта.** Значит ты сам себе senior reviewer. Тот же уровень строгости, что и если бы PR пришёл тебе на ревью от джуна.

---

## Workflow

### Шаг 1 — Re-read изменений

**Не полагайся на память.** Прочитай _все_ файлы которые изменял в этой задаче, целиком, сверху донизу. Даже если помнишь что изменил.

Зачем:
- Ловит leftover debug код (`console.log`, `print`, `debugger`, `TODO`)
- Ловит закомментированный код который забыл удалить
- Ловит случайные изменения не относящиеся к задаче
- Ловит ошибки копипаста

### Шаг 2 — Пройди чеклист ниже

Mental walkthrough по каждому пункту. Нельзя пропускать "потому что очевидно" — привычка проходить чеклист даже когда очевидно — это и есть разница между senior и mid.

### Шаг 3 — Run gates

- `pnpm lint && pnpm typecheck`
- `pnpm build`
- `pnpm test`

Всё зелёное? Идёшь дальше. Что-то красное? Возвращаешься в Execute.

### Шаг 4 — Обнови MEMORY.md / STRUCTURE.md

Если задача повлияла на структуру — обнови `STRUCTURE.md`. Если появились решения / проблемы / known limitations — обнови `MEMORY.md`. Молча, без вопросов.

### Шаг 5 — DONE

Только после всего выше.

## Self-Review Checklist

### 0. Scope

- [ ] Изменил только то что нужно для задачи. Никакого yak-shaving, "заодно отрефакторил".
- [ ] Нет новых файлов сверх необходимого.
- [ ] Нет новых абстракций "на будущее".
- [ ] **Задача доведена до конца, не оставлена на половине.** Нет "TODO: handle later", `raise NotImplementedError`, пустых функций с `pass`, half-implemented веток.
- [ ] **Не использовал human-scale эстимацию** в плане ("неделя", "спринт", "день 1/день 2"). Если думал в таких терминах — значит план неправильный, переделай.
- [ ] Если не успевал — **разделил на независимые подзадачи и выполнил первую полностью**, а не оставил все в half-work состоянии.
- [ ] **Для нетривиальной задачи — research сделан.** Если добавлял новую либу / новую интеграцию / новую фичу — в `.claude/research/{slug}.md` есть отчёт с проверкой existing solutions. Не написал 500 строк того что решается импортом одной библиотеки.
- [ ] **Не изобретал велосипед.** Новые утилиты / хуки / сервисы сверены с `shared/` / `utils/` / `common/` кодбазы и с maintained либами в `package.json` / `requirements.txt` перед написанием.

### 1. Correctness

- [ ] Каждая новая функция делает ровно то что заявлено в имени.
- [ ] Happy path проверен (mental walkthrough или запуск).
- [ ] Error paths обработаны: not-found, duplicate, invalid input, network failure.
- [ ] Race conditions в async коде учтены.
- [ ] Критичные flows (ответы бота по 12 тест-кейсам, повторяемый ingest, голосовые сообщения) не сломаны.

### 2. Types

- [ ] **Backend:** full type hints, mypy strict чистый, нет `Any` без причины.
- [ ] **Frontend:** strict TS, ноль `any`, ноль `@ts-ignore`.
- [ ] Return types явные.
- [ ] Pydantic / TypeScript интерфейсы синхронизированы между фронтом и бэком (если изменил API).

### 3. Architecture

- [ ] Архитектурные границы уважаются (смотри `STRUCTURE.md`).
- [ ] Нет cross-layer import violations.
- [ ] Бизнес-логика в соответствующем слое (не размазана).

### 4. Error Handling

- [ ] Нет `except:` без типа / `catch {}` без обработки.
- [ ] Доменные исключения определены и обработаны.
- [ ] Не глушим ошибки — смотри `.claude/rules/no-fallbacks.md`.
- [ ] Для UI: состояния loading / error / empty / success обрабатываются.

### 5. Tests

- [ ] Написан хотя бы один тест для новой логики.
- [ ] Happy path + error path покрыты.
- [ ] Тесты не зависят от порядка выполнения.
- [ ] Нет `test.skip` / `xit` / `only`.
- [ ] `pnpm test` зелёный.

### 6. Performance

- [ ] Нет N+1 запросов к БД (selectinload / joinedload где нужно).
- [ ] Тяжёлые вычисления мемоизированы где оправдано.
- [ ] Длинные списки виртуализированы если >100 элементов (для UI).
- [ ] Нет рендер-бомб (state меняется каждый тик).

### 7. Security

- [ ] Никаких секретов в коде — всё через `.env` / settings.
- [ ] SQL injection невозможна (ORM / параметризованные запросы).
- [ ] XSS — sanitization на границах.
- [ ] Input validation на внешних границах.

### 8. Code hygiene

- [ ] Нет `TODO`, `FIXME`, `XXX`, `HACK`.
- [ ] Нет `console.log`, `print()`, `debugger`.
- [ ] Нет закомментированного кода.
- [ ] Нет файлов `-old`, `-new`, `-copy`, `.bak`.
- [ ] Имена переменных / функций читаемые.
- [ ] Импорты отсортированы.

### 9. Build & Lint Gates

- [ ] `pnpm lint && pnpm typecheck` — чистый
- [ ] `pnpm build` — прошёл
- [ ] `pnpm test` — зелёный

### 10. Documentation / Memory

- [ ] `STRUCTURE.md` обновлён если структура изменилась
- [ ] `MEMORY.md` обновлён если появились решения / проблемы

### 11. Final Sanity

- [ ] Перечитал diff целиком.
- [ ] Ни один `[ ]` выше не остался пустым (или явно `N/A` с причиной).
- [ ] Если бы этот PR пришёл мне на ревью от джуна — одобрил бы без замечаний.

---

## Если хоть один пункт failed

- Вернись в Execute.
- Почини.
- Снова прогони ВЕСЬ чеклист — не только тот пункт что чинил (могут появиться регрессии).
- Не объявляй DONE пока не пройдёшь весь чеклист с первого захода.

---

## Правило строгости

Если сомневаешься "достаточно ли я проверил" — проверь ещё раз. Цена пропущенного бага в проде > цены 10 минут дополнительного ревью.

**Ты даёшь гарантию, а не оператор нажимающий кнопки.** Твоё имя под каждым релизом.
