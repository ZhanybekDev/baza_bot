# Research — Telegram-бот для застройщика (Часть 1)

**Timestamp:** 2026-07-15
**Task classification:** new-feature + integration (RAG, STT, парсинг)
**Research triggered:** yes

## Summary

База знаний — **~180 KB / ~60k токенов** из **16 источников**. Замеры на живых API показали: наивный RAG с чанкингом по абзацам **проваливает** ключевой тест-кейс, а structure-aware чанкинг с контекстным префиксом попадает точно. Рекомендуемая архитектура — **двухуровневая: intent-роутер → canned-ответы для 6 из 10 групп тест-кейсов, RAG только для контентных вопросов**.

> ⚠️ **Ревизия 2026-07-15 15:50.** Первая версия этого отчёта утверждала «85 KB / 28k токенов, 6 источников» и делала вывод «источников по ДМД2 нет → бот должен отвечать не знаю». **Оба утверждения были неверны.** Причина — читал перечень источников ТЗ невнимательно и пропустил пункт `Общее: — Все презентации проектов: https://disk.yandex.ru/d/BXqSX-BtE_IT2g`. Ниже — исправленные цифры. Промежуточная оценка «83k токенов / 23 документа / 343 MB» из ревью тоже неверна в другую сторону — она включала Бали, дубли и сканы.

По типу источника: `presentation` 397, `kb_table` 82 (xlsx Алисы + docx инструментов продаж), `site` 71.

> Предыдущие замеры RAG шли по индексу в **107 985 симв (59% корпуса)** — в него не попали оба сайта, docx, pptx и «Презентация Бестселлер» (25 827). Сравнение с full-context на 181 600 симв было некорректным. Индексация 550 чанков — **0.04 цента, разово**.

**Важно:** `ЖК ДНК` в старом индексе отсутствовал как проект — его 33 чанка уехали в Бестселлер, потому что проект угадывался по строке пути. Явный реестр это чинит.

## Замеры на живых API (не из доков — реальные запросы)

Вопрос-эталон: «Какая отделка предлагается в ЖК Алиса? Перечисли варианты.»

| Архитектура | Задержка | Цена/запрос | Результат |
|---|---|---|---|
| Full-context (28k токенов в промпт) | 4.5–6.1s | 0.29¢ холодный / **0.08¢ с кэшем** | ✅ ответил верно |
| Наивный RAG (чанки по абзацам, ~139 симв) | 3.7–3.9s | 0.005¢ | ❌ **«не знаю»** — не нашёл |
| **Structure-aware RAG (строка таблицы + префикс)** | **2.8–3.7s** | **0.008–0.011¢** | ✅ ответил верно |

Бюджет ТЗ («не 5 центов и не 30 секунд») выполняют все три. Но structure-aware RAG **в 10–30× дешевле** full-context и вдвое быстрее.

**Почему наивный RAG провалился** — золотой инсайт: «Алиса» это *и ЖК, и колонка Яндекса*. Мелкие чанки без метаданных утонули в шуме про голосового ассистента (top-5 = «Алиса позволит настроить сценарии»). Лечится префиксом `ЖК Алиса (Екатеринбург) — раздел «Отделка» — {заголовок}: {текст}` + фильтром по проекту.

**Дефект, который надо закрыть в промпте:** на вопрос про паркинг модель пересказала «219 м-м» (машиноместа) как «219 м²». Числа и единицы нельзя пересказывать — только цитировать.

## Решающий замер на честном индексе (550 чанков, все 16 источников)

| Вопрос | Full-context (69k ток, 0.70¢) | RAG top-6 (0.013¢) | RAG top-30 + LLM-реранк (0.035–0.045¢) |
|---|---|---|---|
| Отделка **Алиса** | ❌ приплёл «Скандинавию»/«Классик» — это **Бестселлер** | ❌ потерял White Box и Урбан-лофт | ⚠️ Урбан-лофт нашёлся, White Box назван «предчистовая» |
| **ДМД2** этажность/потолки | ✅ «31, 9, 20, 13 эт; 3–3.6 м» | ❌ «со 2 по 16 этаж» | ❌ **регресс** — потолки «не указаны» |
| Отделка **Бестселлер** | ❌ приплёл «Урбан-лофт» — это **Алиса** | ❌ неполно («Всё включено» + White Box) | ✅ **верно** |
| ЖК Сезоны (нет источника) | ✅ не знаю | ✅ | ✅ |

### Что это значит

1. **🔴 Реранк — не решение.** Починил Бестселлер (кейс, валивший section-aware), но **сломал ДМД2**: выбросил из 30 кандидатов чанк с этажностью. Диагноз глубже ранжирования — **нужный фрагмент не доходит даже до top-30**, а реранк не выбирает то, чего нет в кандидатах. Плюс ×3 цена и +2–3s.
2. **🔴 Full-context путает проекты.** На 69k токенов модель смешала отделку Алисы и Бестселлера в обе стороны. Дорого, медленно и врёт в фактах — прямое нарушение требования №4 ТЗ. **Full-context отвергнут не по цене, а по точности.**
3. **✅ Кэш bothub прогревается, но не с первого раза.** 4-й прогон подряд: `cached=69 248`, цена **0.176¢** вместо 0.70¢. Честная цена full-context при потоке — ~0.18¢, разрыв с RAG ~13×, а не 43×. (Замечание C3 ревью подтвердилось.)
4. **Корень проблемы — межпроектный шум.** Все три варианта путают ЖК между собой. Значит первичен **фильтр по проекту**, а не ранжирование.

## Проверенные факты по API

### bothub.ru — ✅ работает
- Base URL: `https://bothub.chat/api/v2/openai/v1`
- **Ключ в PDF разбит переносами строк** — при копировании через OCR/картинку склеивается неверно. Настоящий: `iat=2026-07-02`, `exp=2036`, jti `ZGXOfkeMZuh9aTss`.
- **snake_case работает** (`max_tokens`, `usage.prompt_tokens`) — риска camelCase-несовместимости нет, проверено живым вызовом.
- **Prompt caching работает и не документирован**: 27 904 / 28 025 токенов из кэша, цена ×3.5 вниз. В ответе есть `prompt_tokens_details.cached_tokens`.
- Эмбеддинги: `bge-m3` → **1024 dims**, ctx 8192, $0.0075/1M (в 2.5× дешевле `text-embedding-3-small`, и лучше для русского). Индексация всей базы = **0.018¢ разово**.
- Чат: `gpt-4.1-nano` — $0.075/$0.30 за 1M, 22 токена = $0.0000034.

### Soniox — ✅ ключ живой, полный цикл прогнан
- `stt-async-v5`, русский. Флоу: `POST /v1/files` (multipart, поле `file`) → `POST /v1/transcriptions {model, file_id, language_hints}` → polling `GET /v1/transcriptions/{id}` → `GET /v1/transcriptions/{id}/transcript`.
- **OGG/Opus 48kHz mono принимается напрямую — ffmpeg не нужен.** Проверено на файле, идентичном по формату голосовым Telegram. (Веб-дока утверждает обратное — живой тест бьёт доку.)
- `audio_url` работает, но 🔴 **использовать нельзя**: Telegram-URL содержит **токен бота** → утечка третьей стороне. Грузим байты.
- Замер: 12s аудио → **4.7–5.6s** end-to-end (5 прогонов). Цена **$0.10/час ≈ $0.0003 за голосовое**.
- npm: `@soniox/node` 2.2.0. 🔴 Класс `SonioxNodeClient`, опция **`api_key`** (camelCase молча игнорируется). Пакета `@soniox/sdk` **не существует**.
- 🔴 Ошибки асинхронные: битый вход → `201 queued`, ошибка только при polling. `/transcript` на упавшей задаче → **409**. Проверять `status` обязательно.
- Webhook есть (`webhook_url` + auth-хедер), но при задержке ~5s не нужен — `wait: true` в SDK проще.

### Две ловушки парсинга, найденные живьём (обе молча портят базу)

**1. Tilda `.t-rec` + `innerText` затаскивает инлайновый CSS.** Наивное извлечение блоков дало **128 035 символов** с best.baza.bz при реальном тексте ~9k. Один блок `rec890821720` = **72 107 символов чистого CSS**. Лечится удалением `style,script,noscript` из DOM перед чтением + фильтром только верхнеуровневых `.t-rec` (вложенные дублируют родителя) + дедупом повторяющихся блоков. После фикса: **9 422** (best) и **10 246** (alisa) символов, медиана блока 282/172 симв.
**Бонус:** очищенные `.t-rec` — это готовые смысловые чанки с естественными границами. Сайт Бестселлера прямо отдаёт отделку («Скандинавия», «Классик», предчистовая).

**2. 🔴 Yandex Disk отдаёт имена файлов в NFD — реестр по имени молча теряет источники.** Совпадение по строке провалилось ровно на файлах с буквой **«й»** (`знаний`, `развлечений`): в NFD она раскладывается на `и` + комбинирующий знак. `is_normalized('NFC', name)` → False. Два источника из четырнадцати выпали бы **без единой ошибки**. Лечится `unicodedata.normalize('NFC', …)` на обеих сторонах матча, либо ключом по пути/id вместо имени.

### Сайты — ⚠️ главный сюрприз
- Оба на **Tilda**. Всего 5–6 страниц в sitemap (одностраничники).
- **best.baza.bz защищён антиботом BotFAQtor**: headless Playwright редиректит на `blocked.botfaqtor.ru` (314 симв заглушки вместо 12k контента).
- Обход **штатный, воспроизводим 3/3**: bundled chromium + `headless: false` → 141 Tilda-блок, 9.4k символов.
- `channel: 'chrome'` заблокирован даже в headed. `--headless=new` заблокирован.
- **Следствие для деплоя:** парсер обязан идти через `xvfb-run` в Docker, headless не годится.
- alisa.baza.bz антибота не имеет, 133 блока.

## 🔴 Ловушка в тест-кейсах (важно для созвона)

Таблица вознаграждений перечисляет **11 ЖК**, а источники даны **только по двум**: Бестселлер (Москва) и Алиса (Екатеринбург). Тест-кейс **«Расскажи про ЖК ДМД2?»** — источников по ДМД2 **нет**.

→ Правильный ответ бота = честное «не знаю» + предложение менеджера. Это не пробел, а **проверка** требования №4 ТЗ. Бот, который сочинит описание ДМД2, проваливает тест.

## Recommendation

**Подход:** двухуровневая гибридная архитектура.
1. **L0 — роутер:** эмбеддинг вопроса (bge-m3) + keyword-буст → интент.
2. **L1 — canned:** 6 групп отдаются из БД дословно, **без LLM**. ~0.00002¢, ~1.5s (только embed).
3. **L2 — RAG:** structure-aware чанки с префиксом + фильтр по проекту → top-k → `gpt-4.1-nano` с жёстким «цитируй, не пересказывай числа».

**Почему не full-context:** дороже в 10–30× на запрос, вдвое медленнее, и — главное — нечем показать источник ответа в админке (Часть 2 требует «что лежит в базе знаний»).

**Почему не наивный RAG:** замер показал провал на реальном тест-кейсе.

**Не делаем:** RRF (не оправдан на 500 чанках), HNSW-индекс (объём 2 МБ), FTS с русской морфологией (0 recall на перефразировках).
