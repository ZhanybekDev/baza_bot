# Plan: Telegram-бот-ассистент для агентов застройщика (Часть 1)

**Goal:** Живой Telegram-бот, отвечающий партнёрам-риелторам по базе знаний BAZA Development — текстом и голосом, с дословным цитированием цифр, честным «не знаю» и базой, обновляемой одной командой из публичных ссылок ТЗ.

**Context:** 17 источников, ~560 чанков, 6 ЖК с материалами, 11 проектов в реестре. Архитектура выбрана по четырём сравнительным замерам на живых API. Главный урок восьми ревью: **таблица тест-кейсов — это техзадание, а не контент**. Из неё берутся триггеры, факты, инструкции и эталоны — но не формулировки.

**Total estimate:** 7–8.5 часов / 2 сессии Claude Code

**Стек:** Nuxt 4.4.8 + TypeScript + Prisma 7.8 (+`@prisma/adapter-pg`) + Postgres 17 + pgvector, grammY 1.44, Playwright 1.61, `@soniox/node` 2.2.0, bothub. Деплой — домашний сервер Ubuntu 24.04 + Docker Compose + существующий Cloudflare Tunnel.

## Как читается таблица тест-кейсов (это определяет всю Phase 2)

Колонка «Пример ответа» смешивает четыре жанра. Импортировать её дословно нельзя — ровно это ловило ревью v8.

| Жанр | Строки | Куда идёт |
|---|---|---|
| **Факты** — можно цитировать | Вознаграждение (таблица %), Ипотека (ссылка) | В `CannedAnswer` как данные |
| **Заметка разработчику** — переписать | Сделка («*я думаю, надо переводить на КЦ*»), Фиксация («*раздлеле… (ссылка в аппке)*») | Автор пишет текст, ячейка — спека |
| **Инструкция боту** — в промпт/роутер | Характеристики ЖК («*Нужно продуктово отвечать так: корпуса, очереди, этажность, тип строительства, количество квартир, коэффициент парковочных мест, высота потолков, формат окон, балконы*»), Справочник («*Не учитывай регистр*») | В system-промпт и реестр |
| **Образец стиля** — не список фактов | Отделка (419 симв: **только White Box + Классика**, текст оборван на полуслове) | Референс тона и структуры ответа |

🔴 **Эталон заказчика ≠ полный ответ.** В его 419 символах **два** варианта отделки, а в базе знаний Алисы (лист «Отделка») — **три**: White Box, Классика, **Урбан-лофт**. Ячейка обрывается на «Стены — обои под покраску.» Это иллюстрация формата («вот так надо отвечать»), а не исчерпывающий перечень. **Источник истины по фактам — база знаний, не ячейка.** Тест сверяет с эталоном тон и структуру (нумерованный список, название + характеристики), а полноту — по базе.

🔴 **`CannedAnswer` содержит авторский выверенный текст + ссылку на строку-источник.** Сидер не импортирует ячейку — он падает, если у группы нет выверенного текста.

**Решения (не трогать при ревью):**
- **Фиксация → ссылка на партнёрскую страницу региона** (`/ekb` или `/msk`), где живёт кнопка «Закрепить клиента». Решение владельца. Глубокую ссылку на CRM-форму `bazaforms.bitrix24.site/crm_form422/` **не даём**: она озаглавлена «ЕКБ — заявка от Риэлтора», а `/msk` ссылается на неё же — похоже на баг сайта заказчика, транслировать его не будем.
- **«Яна Крылосова»**, не «Крылосовла» из таблицы. Решение планировщика по двум независимым подтверждениям: официальная страница `/ekb` в блоке «Контакты» пишет «Яна Крылосова», и её телеграм `@yanakrylosovaaa` содержит **krylosova**. Таблица тест-кейсов содержит россыпь опечаток («раздлеле», «отдлека», «Москоский», «деатльно») — «Крылосовла» из того же ряда. Неверная фамилия живого человека в ответе агенту — заметный дефект.

## Архитектура (идёт в README)

```
вопрос
 ├─ 1. интент (keyword fast-path, триггеры из таблицы) ─→ авторский canned, 0 LLM
 │     ⚠️ canned выбирается С УЧЁТОМ региона проекта из вопроса
 ├─ 2. alias-роутер (регистронезависимо, по справочнику)
 │   ├─ класс = агентство → «BAZA реализует как агентство, КВ 1%» + менеджер ЕКБ
 │   ├─ класс = Бали → вне области бота + менеджер
 │   └─ есть источники → ВЕСЬ корпус проекта → gpt-4.1-nano
 └─ 3. проект не назван → pgvector top-k по всей базе
```

**Приоритет: интент > проект** — 3 из 12 тест-кейсов содержат оба, везде побеждает canned.

**Замеры, 4 архитектуры на честном индексе (550 чанков, 16 источников):**

| Вопрос | Full-context 69k (0.70¢) | RAG top-6 (0.013¢) | RAG+реранк (0.04¢) | **Project-scoped** |
|---|---|---|---|---|
| Отделка Алиса | ❌ приплёл «Скандинавию» (Бестселлер) | ❌ потерял White Box, Урбан-лофт | ⚠️ частично | ✅ **все три** |
| ДМД2 этажность | ❌ **выдумал потолки** | ❌ «со 2 по 16 этаж» | ❌ регресс | ✅ «31, 9, 20, 13» |
| Отделка Бестселлер | ❌ приплёл «Урбан-лофт» (Алиса) | ❌ неполно | ✅ | ✅ **верно** |
| ЖК Сезоны | ✅ | ✅ | ✅ | ✅ **0 LLM** |

- **Full-context отвергнут по точности:** на 69k смешивает проекты. На ДМД2 выдал «потолки 3–3.6 м» — цифры в ДМД2 нет, она из презентации **ЖК ДНК**. ТЗ: «выдуманные цены — сразу минус».
- **Наивный RAG:** 3 провала из 4. Scores сбиты в кучу (0.753 против 0.688), соседние разделы вытесняют нужный.
- **Реранк:** починил Бестселлер, сломал ДМД2 — нужный чанк не доходит до top-30. ×3 цена, +2–3s.
- **Семантический роутер:** верный проект 0.557 против ложного 0.527 — зазор 0.03.
- **Project-scoped:** корпус одного ЖК мал, retrieval внутри не нужен; роутинг детерминирован справочником.

**Цена и задержка (3 прогона, без кэша):**

| Ветка | prompt_tokens | цена | задержка |
|---|---|---|---|
| canned / агентский / Бали | 0 | **0** | ~0 |
| малый проект (ДМД1/ДМД2/ДНК/КП) | ~3.5–6.5k | 0.017–0.07¢ | **2.28–3.01s** |
| Алиса | 20 398 | 0.053–0.205¢ | **2.51–3.90s** |
| Бестселлер | 25 675 | 0.069–0.259¢ | **2.46–5.28s** |

**Худший случай: 0.259¢ и 5.28s** при лимитах 5¢ и 30s — запас 19× и 6×. ⚠️ Кэш bothub недетерминирован (`0 / 20 224 / 0`) — не закладываемся.

## Git-процесс (требование ТЗ)

ТЗ: «репозиторий на GitHub с инструкцией запуска — обоснованием архитектуры — **нормальной историей коммитов. Один коммит „init“ — плохой знак**». Проект не инициализирован.

- `git init` + приватный репо + доступ проверяющему — до Phase 1.
- Ветка на фазу (`feat/ingest`, `feat/bot-core`, `feat/voice`), squash-merge в `main`. Фазы >2 файлов — в worktree.
- Атомарные коммиты по шагам outline. Pre-commit: `lint && typecheck && test`; `build` до пуша. **Без `Co-Authored-By`**, Conventional Commits.
- В репо: `CLAUDE.md`, `.claude/{plans,research,reviews}/`. Секреты — только `.env.example`.
- **README — три блока:** (1) инструкция запуска, воспроизводимая на чистой машине; (2) обоснование архитектуры с таблицей замеров; (3) что не вошло (Бали, OCR-сканы) и почему.

## Phase 1: Реестр проектов + повторяемый ingest

**Estimate:** 3–3.5 часа

**What changes:**
- `docker-compose.yml` — `pgvector/pgvector:pg17`, хост-порт **5433** (5432 занят локальным `postgresql@18`)
- `prisma/schema.prisma` — `Project` (name, aliases[], region, kind, **fee: {base, bonus, bonusRule, source}**), `Source`, `Document`, `Chunk`, `IngestRun`
- `prisma/migrations/*_init/migration.sql` — `CREATE EXTENSION vector` + `ADD COLUMN embedding vector(1024)` руками
- `server/db/client.ts` — Prisma через `@prisma/adapter-pg`, `import "dotenv/config"`
- `server/registry/projects.ts` — 11 проектов из справочника: алиасы, регион, класс, КВ
- `server/registry/sources.ts` — **17 фиксированных источников** (16 базовых + гуглдок «Возражения» → Бестселлер/МСК)
- `server/ingest/fetchers/portal.ts` — **обход дерева `realtors.baza.bz`** от `/ekb` и `/msk` по внутренним ссылкам и `link=` из Tilda-таблиц. **Не фиксированный список URL** — деревья несимметричны и меняются. Даёт ~15 страниц → итого **~32 источника**
- `server/ingest/fetchers/{site,gdocs,yadisk}.ts`, `parsers/{xlsx,docx,pptx,pdf}.ts`, `{chunk,embed,run}.ts`
- `Dockerfile.ingest` — chromium + xvfb
- `.env.example`, `README.md` (инструкция запуска ingest)

**Ключевые решения (проверены живьём):**
- **Проект — из реестра, не из пути.** Прототип ошибся: `презентация_днк.pdf` уехал в Бестселлер, ЖК ДНК исчез. Цена: ДМД1 2.5% против ДМД2 2.7%.
- 🔴 **NFC-нормализация.** Yandex Disk отдаёт NFD — матч провалился ровно на файлах с «й» (`знаний`, `развлечений`); два источника выпали бы молча.
- 🔴 **Tilda: выпилить `style/script` до `innerText`.** Наивно — **128 035 символов** вместо 9 422; один блок = 72 107 символов CSS. Только верхнеуровневые `.t-rec` + дедуп. После фикса блоки — готовые чанки (медиана 282 симв).
- **Headed chromium** — best.baza.bz отдаёт headless заглушку BotFAQtor (314 симв); headed проходит 3/3.
- 🔴 **Портал троттлит: 403 при быстрых запросах.** Обход curl'ом без пауз упёрся в блок со второй страницы — при том что одиночный запрос отдаёт 200. Обход портала: headed-браузер + **пауза ~1s между страницами**. Без этого ingest молча соберёт пустые страницы.
- **Ingest из URL**, не из `~/Downloads` — иначе запрещённая ТЗ копипаста. Гуглдок «Возражения» доступен (`/export?format=docx` → 200).
- **Сканы — громко в `needs_ocr`:** `Инфраструктура_ДНК.pdf` (5 симв / 6 стр), `Презентация_мобильная.pdf` (32 / 33), `Отделка.pdf` МСК (1250 / 26).
- **Дедуп по content-hash:** `Презентация Бестселлер.pdf` приходит дважды, байт-в-байт.
- **Без векторного индекса:** ~560 × 1024 × 4B ≈ 2.3 МБ, точный скан.

**Acceptance criteria:**

- [ ] `fee` Алисы = `{base: 2.6, bonus: 1, bonusRule: "или в вознаграждение, или в скидку клиенту"}`, источник — `/ekb/svod`; Бестселлер = 2.1 с `/msk`

## Phase 2: Ядро ответа + живой бот в Telegram

**What changes:**

**Ключевые решения:**
- 🔴 **Canned-тексты пишутся, а не импортируются.** Ячейка — спека. «Сделка» → «По вопросам сделки, документов и брони — контакт-центр 305-05-05» (а не «я думаю, надо переводить»). «Фиксация» → ссылка на партнёрскую страницу региона, где кнопка «Закрепить клиента». Сидер **падает**, если у группы нет выверенного текста.
- 🔴 **Canned выбирается по региону проекта:** рассрочка и фиксация → `/ekb` или `/msk`; вознаграждение → таблица для ЕКБ/агентских, **2,1%** для Бестселлера; менеджер → по региону.
- **Продуктовый шаблон в промпт:** на «Расскажи про ЖК X» отвечать по структуре заказчика — корпуса/очереди/этажность, тип строительства, количество квартир, коэффициент парковочных мест, высота потолков, формат окон, балконы, особенности. **Только те пункты, что есть в источнике** — отсутствующие пропускаются, не выдумываются.
- **Числа цитируются дословно.** Замеры поймали: «219 м-м» → «219 м²», и потолки ДМД2 из чужого ЖК.
- **«Крылосовла Яна»** — дословно как в источнике (решение владельца).
- **Prompt caching не проектируем** — недетерминирован.

**Acceptance criteria:**
- [ ] Бот у @BotFather, токен в `.env`, `setWebhook` с secret token; живой по публичной ссылке
- [ ] **Прогон по 12 вопросам таблицы**, зафиксирован тестом
- [ ] Ни один canned-ответ **не содержит мета-речи** источника — тест на подстроки «я думаю», «надо переводить», «Нужно продуктово отвечать», «ссылка на этой раздел», «раздлеле»

## Phase 3: Голосовые сообщения

**Estimate:** 1–1.5 часа

**What changes:**
- `server/core/stt.ts` — Soniox `stt-async-v5` через `@soniox/node`
- `server/api/telegram.post.ts` — хендлер `message:voice`
- `prisma/schema.prisma` — `Message.kind`, `Message.transcript`
- `README.md` — секция STT

**Ключевые решения (проверено живым прогоном):**
- **ffmpeg не нужен** — Soniox принимает OGG/Opus 48kHz mono напрямую.
- 🔴 **Байты через `/v1/files`, НЕ `audio_url`** — Telegram-ссылка содержит **токен бота**; отдать её Soniox = отдать бота.
- 🔴 Опция **`api_key`**, не `apiKey` (camelCase молча игнорируется). Класс `SonioxNodeClient`. `@soniox/sdk` не существует.
- 🔴 Ошибки асинхронные: битый вход → `201 queued`; `/transcript` на упавшей задаче → **409**. Проверять `status` обязательно.
- `language_hints: ['ru']`, `wait: true`. Замер: 12s аудио → **4.7–5.6s**, **$0.0003**.

**Acceptance criteria:**
- [ ] Голосовое на русском → транскрипт → тот же ответ, что на текстовый аналог
- [ ] «какой процент КВ по Алисе» голосом отрабатывает end-to-end
- [ ] **Токен бота не покидает наш сервер**
- [ ] Транскрипт в БД; полный цикл **< 10s**, в README
- [ ] Битое аудио → внятная ошибка из `error_type`, без 409 и падения
- [ ] Unit: мок Soniox + ветка `status=error`. Integration: живой ogg
- [ ] Несколько атомарных коммитов в `feat/voice`; `pnpm build && lint && typecheck` зелёные

**Implementation outline** (шаг = атомарный коммит в `feat/voice`):
1. Клиент Soniox: `getFile` → скачать байты → `/v1/files` → `transcribe({file_id, wait:true})` → проверка `status`
2. Хендлер `message:voice` → то же ядро ответа, что и текст
3. Транскрипт + обработка `status=error` → squash-merge

**Test strategy:**
- Unit: клиент на моках, ветка `status=error`, отказ при пустом транскрипте
- Integration: реальный ogg → транскрипт → ответ ядра
- Manual: голосовое в Telegram, сверка с текстовым аналогом

## Out of scope

- **Бали** (Origins, Kedungu) — **есть в справочнике заказчика с алиасами**, исключены решением владельца: бот адресован агентам по российским ЖК, в таблице вознаграждений их нет. Остаются в реестре с классом `Бали` → бот распознаёт вопрос и отвечает, что это вне его области, + менеджер.
- **OCR** — 3 скана в `needs_ocr`; их содержание дублируется другими источниками.
- Админка (Часть 2), страница из Фигмы (Часть 3), авторизация агентов.

## Risks / Unknowns

- **grammY в Nitro** — адаптера нет, связка не подтверждена докой. → Первый шаг Phase 2; фолбэк — long polling.
- **Опечатки вне справочника** («отдлека», «Алсиа») — регулярки не ловят. → Матч по основам + pgvector fallback. Fallback проверен: «Какая отделка в Милом доме?» → глобальный top-6 сам сгруппировался вокруг ДМД1/ДМД2 и ответил верно.
- **Домашний ПК** — пользователь подтвердил, что не выключается. Остаточный риск: **Tailscale не поднят**, удалённо чинить нельзя. → Поднять до защиты.
