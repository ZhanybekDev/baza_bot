## Critical Findings

### C1. Нет шага получения токена бота
- **Dimension:** D6 (Failure Mode) / Missing Step
- **Where:** Phase 2 целиком
- **Problem:** ТЗ требует **ссылку на работающего бота в Telegram**. План нигде не заводит бота у @BotFather и не упоминает `BOT_TOKEN` иначе как в `.env.example`. Без токена Phase 2 не имеет выхода.
- **Recommendation:** явный шаг 0 в Phase 2: создать бота, положить токен в `.env`, задать `setWebhook` с secret token.
- **Applied:** No — BLOCK.

### C2. Деплой запланирован, но VPS не подтверждён
- **Dimension:** D6 (Failure Mode)
- **Where:** Phase 2, шаг 6 «Деплой на VPS: Caddy + HTTPS»
- **Problem:** Пользователь выбрал «VPS + Docker Compose», но **наличие VPS, домена и доступа не проверялось**. Caddy требует домен с валидным A-записью для ACME. Telegram webhook требует публичный HTTPS с доверенным сертификатом. Если VPS нет — Phase 2 не закрывается, а это фаза с главным deliverable.
- **Recommendation:** подтвердить VPS+домен до старта Phase 2, либо переключить бота на long polling (тогда HTTPS нужен только админке, Часть 2).
- **Applied:** No — BLOCK.

### C3. Prisma 7 требует driver adapter — в плане его нет
- **Dimension:** D1 / Missing Step
- **Where:** Phase 1, «What changes» → `prisma/schema.prisma`
- **Problem:** В Prisma 7 Rust-движок убран, QueryCompiler — GA, **драйвер-адаптеры обязательны**. План не упоминает `@prisma/adapter-pg` (существует, 7.8.0), не упоминает ESM-only и то, что env больше не подхватываются сами (`import "dotenv/config"`).
- **Recommendation:** добавить `@prisma/adapter-pg` в зависимости и инициализацию клиента через адаптер.
- **Applied:** No — BLOCK.

### C4. Часть источников — сканы без текстового слоя
- **Dimension:** D10 (Input Quality Audit)
- **Where:** Phase 1, парсеры
- **Problem:** Замер показал, что текстовый слой есть не везде:
  - `Инфраструктура_ДНК.pdf` — **5 символов на 6 страниц**
  - `Презентация_мобильная.pdf` — **32 символа на 33 страницы**
  - `Отделка.pdf` (МСК) — 1 250 симв на 26 страниц (подозрительно мало для документа про отделку)

  План молча предполагает, что `pdf.ts` извлечёт текст отовсюду. Для этих файлов ingest тихо запишет пустоту — **классический silent degradation, прямое нарушение NO FALLBACKS** из user-memory.
- **Recommendation:** порог в ingest — если извлечённый текст < N символов на страницу, документ помечается `needs_ocr` и **громко падает в лог/статус**, а не проглатывается. Решение про OCR (или явное исключение) принять отдельно.
- **Applied:** No — BLOCK.

### C5. Дубликат источника → двойные чанки
- **Dimension:** D5 (Input/Output Waste)
- **Where:** Phase 1, идемпотентность по sha256
- **Problem:** `Презентация Бестселлер.pdf` присутствует и среди 6 исходных источников, и в папке (`/МСК/Презентации/`). Замер обоих даёт **85 стр / 25 827 симв — совпадение точное**. Хеш считается от сырого текста *документа*, а не от чанка, поэтому два источника с одинаковым содержимым дадут **два комплекта идентичных чанков**. Ретривер получит дубли в top-k → в контекст уедет один и тот же текст дважды, вытеснив полезное.
- **Recommendation:** дедуп по content-hash **до** чанкинга, с приоритетом источника.
- **Applied:** No — BLOCK.

### C6. Canned-путь платит 1.5s за embed, которого можно избежать
- **Dimension:** D3 (Cost/Resource Profile)
- **Where:** Phase 2, схема «вопрос → embed → L0 роутер → L1 canned»
- **Problem:** План декларирует canned как «ноль LLM» — это правда, но **не ноль сети**. Каждый canned-ответ всё равно ждёт эмбеддинг (замерено 1.45–1.78s), и это **вся** его латентность. Для точных триггеров из таблицы («КВ», «АВ», «комиссия», «рассрочка») хватит нормализации + словарного матча → ~0 мс, ноль внешних вызовов.
- **Recommendation:** keyword fast-path перед embed; embed только если словарь не дал уверенного попадания. Побочный бонус — чинит промах роутера «ипотека под 6%» → рассрочка (11/12).
- **Applied:** No — BLOCK.

### C7. Идемпотентность по sha256 может не срабатывать на Tilda
- **Dimension:** D6 (Failure Mode)
- **Where:** Phase 1, «Идемпотентность через content hash»
- **Problem:** Хеш берётся от текста, снятого с живой страницы. Tilda-страницы содержат динамику (счётчики, даты, A/B-блоки, `?t=` в ассетах). Если хоть один символ плавает от прогона к прогону — хеш меняется **всегда**, оптимизация «не переэмбеддим» не срабатывает никогда, и каждый `pnpm ingest` жжёт полный переэмбеддинг. Не фатально (66s / 0.018¢), но заявленная в плане оптимизация — мнимая.
- **Recommendation:** хешировать **нормализованный** текст (убрать числа-счётчики/таймстемпы) либо честно признать, что для сайтов хеш не экономит, и считать его только для документов.
- **Applied:** No — BLOCK.

### C8. Расхождение версий Postgres
- **Dimension:** D2 (Cross-Component Pipeline)
- **Where:** Phase 1, `docker-compose.yml` → `pgvector/pgvector:pg17`
- **Problem:** План фиксирует pg17, а на машине уже стоит и запущен `postgresql@18` (18.3, launch agent, поставлен research-агентом). Два Postgres на 5432 = конфликт порта при `docker compose up`. Плюс в PG18 `STORED` для generated-колонок перестал быть дефолтом — если позже появится FTS-колонка, поведение разойдётся между локалью и Docker.
- **Recommendation:** либо остановить локальный (`brew services stop postgresql@18`), либо поднять Docker на 5433. Зафиксировать версию явно.
- **Applied:** No — BLOCK.

## Verdict Rationale

Три BLOCKING-находки, все из одного корня: **план проектировался на неполном прочтении источников ТЗ**. Пропущен целый пункт раздела «Источники для базы знаний» — папка с 23 документами, включая единственные материалы, прямо адресованные аудитории бота (презентация для агентов, регламент работы с партнёрами).

Отягчающее: план не просто упустил данные, он **возвёл их отсутствие в критерий приёмки** (B2). Это не пробел, а инверсия — реализация по плану гарантированно провалит тест-кейс заказчика на защите.

Архитектурное ядро (двухуровневый роутер + structure-aware RAG, headed-парсинг, байты в Soniox вместо URL с токеном) **остаётся верным** и от новых данных только выигрывает. Но фундамент под обоснованием — цифра 28k токенов — неверен в 2.6×, а обоснование в README ТЗ называет важной частью оценки. Патчить это точечно нельзя: меняются входы, объём, число чанков, список покрытых ЖК, замеры и acceptance criteria двух фаз из трёх. По правилу строгости — `BLOCK`.

### C9. План закладывает Caddy — на этом сервере он не нужен и конфликтует
- **Dimension:** D8 (Abstraction Level) + D12 (Prior Art / Build vs Adopt)
- **Where:** Phase 2 → `Caddyfile`, `docker-compose.prod.yml`; шаг 6 «Деплой на VPS: Caddy + HTTPS»
- **Problem:** План проектировался под абстрактный VPS и вводит Caddy ради авто-HTTPS через ACME. На реальном сервере:
  1. **HTTPS уже решён** — Cloudflare Tunnel терминирует TLS, сертификаты не наши забота;
  2. **порт 80 занят** контейнером `other-app-1-frontend` → Caddy на 80 не поднимется, ACME HTTP-01 не пройдёт;
  3. домен `example.com` живёт в Cloudflare, DNS-01 тоже избыточен.
- **Evidence:** `/etc/cloudflared/config.yml`:
  ```yaml
  ingress:
    - hostname: other-app-1.example.com
      service: http://localhost:80
    - hostname: other-app-2.example.com
      service: http://localhost:8090
    - hostname: other-app-2-api.example.com
      service: http://localhost:3001
    - service: http_status:404
  ```
  Плюс раздел 7 инструкции прямо предусматривает сценарий «Добавить новый проект на поддомен».
- **Recommendation:** выкинуть Caddy. Публикация = 2 строки в `ingress` (`baza-bot.example.com` → `http://localhost:<free>`), CNAME в Cloudflare, `systemctl restart cloudflared`. Порт брать свободный (8091/3002 — 80/3001/8090 заняты). Telegram webhook получает валидный HTTPS из туннеля.
- **Applied:** No — BLOCK.

### C10. Сервер — домашний ПК; это риск для главного deliverable
- **Dimension:** D6 (Failure Mode Analysis)
- **Where:** Phase 2, acceptance «Бот живой по публичной ссылке»
- **Problem:** Инструкция прямым текстом: *«Сервер — домашний ПК под Ubuntu»*, *«Нельзя надолго отключать питание и LAN-кабель»*. ТЗ требует **работающую ссылку на бота**, которую проверяющий откроет в произвольный момент, и созвон-защиту на 30–40 минут. Выключенный/перезагруженный ПК = мёртвый бот в момент оценки. Это не гипотетика: Tailscale уже сейчас недоступен, то есть удалённо починить в случае чего не выйдет.
- **Recommendation:** осознанное решение пользователя, не техфикс. Варианты: (а) принять риск, включить Tailscale для удалённого доступа и не выключать ПК до защиты; (б) арендовать VPS на время оценки. Зафиксировать выбор до Phase 2.
- **Applied:** No — требует решения пользователя (см. U5).
