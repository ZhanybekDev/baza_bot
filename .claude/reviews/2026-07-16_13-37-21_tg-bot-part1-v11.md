## BLOCKING Findings

### B1. Комиссия Алисы: живой источник заказчика противоречит таблице тест-кейсов
- **Dimension:** D10 (Input Quality Audit)
- **Where:** Реестр → «Алиса … КВ **2.7%**»; canned «Вознаграждение» → таблица из тест-кейсов
- **Problem:** `/ekb/svod` — «Свод по проектам BAZA Development в Екатеринбурге», живая страница партнёрского портала:
  ```
  Свод по проектам ; «Алиса» ; «Милый дом» оч.2 ; «Милый дом» оч.1 ; «ДНК» ; Проекты ПИФ
  Агентская комиссия ; 2,6% - 3,6%** ;      2,7% ;         2,5% ;   3,5% ;          1%
  ```
  По ДМД1, ДМД2, ДНК и ПИФ цифры совпадают с таблицей. **По Алисе — нет**: таблица говорит 2.7%, портал — **2,6% - 3,6%**.
- **Почему BLOCKING:** план захардкодил 2.7% и объявил canned-таблицу источником истины по вознаграждениям. ТЗ: *«Выдуманные цены и условия — сразу минус»*. Процитировать 2,7% там, где заказчик публикует 2,6%-3,6%, — ровно этот случай. Плюс «какой процент КВ по Алисе» — тест-кейс из таблицы, его будут гонять на защите.
- **Fix required:** источник комиссий — `/ekb/svod` (живой), таблица тест-кейсов — фолбэк. Конфликт по Алисе вынести владельцу (см. U1).

### B2. Комиссия — не число, а механика; план потерял половину смысла
- **Dimension:** D10 / D1
- **Where:** `Project.feePercent` — скалярное поле
- **Problem:** Сноска `**` на `/ekb/svod` дословно:
  > Агентам дополнительно предоставляется в распоряжение сумма в размере **1% от стоимости квартиры**, которая может быть использована: — или в качестве **увеличения агентского вознаграждения на 1%**; — или в качестве **дополнительной скидки для клиента в размере 1%**. **Может быть выбран только один** из предложенных вариантов.

  Это ключевая для агента механика: 2,6% базовых + 1% на выбор — себе или клиенту. `feePercent: number` её не выражает. Бот, отвечающий «2,6%», умалчивает о половине предложения; отвечающий «3,6%» — вводит в заблуждение.
- **Fix required:** `feePercent` → структура (base, bonus, bonusRule) либо текстовое поле с оговоркой. Ответ обязан содержать механику выбора.

### B3. Партнёрский портал — дерево, а не две страницы
- **Dimension:** D10
- **Where:** `server/registry/sources.ts` → «19 источников: … + `/ekb` и `/msk`»
- **Problem:** Живые и релевантные, все HTTP 200:

  | URL | Содержимое |
  |---|---|
  | `/ekb/svod` | Сводная таблица: комиссии + матрица программ по 5 проектам |
  | `/ekb/ipoteka` | Ипотечные программы ЕКБ |
  | `/ekb/rassrochka` | Условия рассрочки (1574 симв) |
  | `/ekb/treyd-in` | Трейд-ин |
  | `/ekb/sk-mesyaca` | Скидка месяца / спецпредложения |

  План знает только корневые `/ekb` и `/msk` — то есть оглавления. Содержание — на дочерних. `/msk/svod` → **404**, у Москвы дерева нет (комиссия 2,1% дана прямо на `/msk`) — асимметрия, которую ingest должен пережить.
- **Fix required:** фетчер портала обходит дерево от `/ekb` и `/msk` по внутренним ссылкам (как sitemap-обход), а не тянет два фиксированных URL. Источников станет ~24.

## Verdict Rationale
