## Findings by Dimension

### D11. Output Verification — главный источник findings

**C1. Alias-роутер не переживает русские падежи**
- **Where:** Phase 2 → `server/core/router.ts`, словарь алиасов
- **Problem:** Регулярка `\bмилый дом\b` не ловит склонённые формы. Прогон:

  | Форма | Результат |
  |---|---|
  | «Расскажи про Милый дом» | ✅ |
  | «Какая отделка в **Милом доме**?» | ❌ промах |
  | «Что по **Милому дому**?» | ❌ промах |
  | «планировки **Милого дома**» | ❌ промах |
  | «Милый дом 2 очередь» | ✅ |

  **3 формы из 5.** Причём промах взят не из головы — вопрос «На какой срок можно взять рассрочку **в Милом доме**» стоит в таблице тест-кейсов заказчика дословно.
- **Почему не BLOCKING:** деградация мягкая — промах уводит в fallback (глобальный top-6), который **эмпирически отработал верно** на этих же вопросах (top-6 сам сгруппировался вокруг ДМД1/ДМД2, ответ про этажность точный). То есть это потеря точности роутинга, а не потеря ответа.
- **Fix applied:** в план добавлено требование матчить по основам (стемминг/`pymorphy`-подобная нормализация или паттерны вида `мил\w*\s+дом\w*`) + регрессионный тест на склонённые формы.

**C2. Конфликт «проект + интент» не специфицирован и не покрыт тестом**
- **Where:** Phase 2 → схема архитектуры, acceptance criteria
- **Problem:** **3 из 12** тест-кейсов содержат одновременно и проект, и интент:
  - «Сколько первоначалка на **ДМД2**» → проект ДМД2 + интент рассрочка
  - «Какая рассрочка на **Алису** и какой там ПВ» → Алиса + рассрочка
  - «какой процент КВ по **Алисе**» → Алиса + вознаграждение

  По таблице заказчика во всех трёх побеждает **интент** (рассрочка → ссылка на `realtors.baza.bz/ekb`; КВ → полная таблица % по 10 ЖК). План рисует fast-path первым в диаграмме — то есть проектирует правильно, — но **приоритет нигде не сформулирован явно и не проверяется тестом**.
- **Отягчающее:** мой собственный замер E прогнал «Сколько первоначалка на ДМД2» **без** fast-path, получил project-scoped → «в базе не указано» + менеджер, и я отрапортовал это как «✅ роутинг сработал». Фактически это **неверный ответ** — правильный ответ канонический canned. Валидация архитектуры ни разу не проверила порядок.
- **Fix applied:** в план внесено явное правило приоритета (интент > проект) + acceptance-тест на все 3 конфликтных кейса + требование, чтобы триггеры рассрочки включали «первоначалка», «ПВ», «срок рассрочки» дословно из таблицы.

**C3. Главный риск плана — артефакт замера**
- **Where:** Risks → «Задержка Алисы 21.9s на холодном кэше — единственный замер вне комфортной зоны»
- **Problem:** Не воспроизводится. 3 прогона × 3 проекта с корректным таймингом:

  | Проект | задержка | цена |
  |---|---|---|
  | Алиса (191 чанк, 20 398 ток) | **2.51–3.90s** | 0.053–0.205¢ |
  | Бестселлер (199, 25 675 ток) | **2.46–5.28s** | 0.069–0.259¢ |
  | ДМД2 (37, 5 577 ток) | **2.28–3.01s** | 0.017–0.058¢ |

  Объяснение «холодный кэш» опровергается данными: первый прогон Алисы был холодным (`cached=0`) и занял **3.90s**, а не 21.9s. Плюс Бестселлер с **бо́льшим** числом токенов давал 4.15s в том же замере — 21.9s противоречил собственным соседним данным ещё тогда.
- **Fix applied:** риск заменён на реальные цифры; митигация «резать корпус Алисы по разделам» снята как решение несуществующей проблемы.

### D3. Cost / Resource Profile

**C4. Кэш срабатывает через раз — митигация на него опираться не может**
- **Where:** «Кэш bothub работает и прогревается», Phase 2 → «Prompt caching по проекту», Risks → «Прогрев кэша при старте»
- **Problem:** Кэш недетерминирован: Алиса — `cached=0 / 20 224 / 0` по прогонам 1-2-3; Бестселлер — `0 / 0 / 25 472`. То есть попадание не гарантировано даже на идентичном префиксе подряд. Цена гуляет в 4 раза (0.053↔0.205¢).
- **Почему не BLOCKING:** и **без** кэша всё в бюджете — худший случай 0.259¢ при лимите 5¢, 5.28s при лимите 30s. Кэш — приятный бонус, а не несущая конструкция.
- **Fix applied:** формулировки «кэш прогревается» и «прогрев кэша при старте» заменены; в README цифры даются по **худшему случаю без кэша**.

### D2. Cross-Component Pipeline

**C5. «Милый дом» без очереди молча уходит в ДМД1**
- **Where:** словарь алиасов, паттерн `\bдмд\b|\bмилый дом\b` → ДМД 1 очередь
- **Problem:** Вопрос «Расскажи про Милый дом» без указания очереди детерминированно уводит в ДМД1, а ДМД2 молча теряется. Цена ошибки прямая: вознаграждение **2.5% у 1 очереди против 2.7% у 2-й**, сроки и условия разные. Это тихая подмена, а не ошибка — агент не узнает, что получил ответ только по половине проекта.
- **Fix applied:** правило — при «Милый дом» без очереди отдавать **обе очереди** с явной маркировкой, а не выбирать одну.
